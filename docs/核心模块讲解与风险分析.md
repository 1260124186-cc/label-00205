### 1. 数据预处理模块

**文件位置**: [preprocessing.py](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/services/preprocessing.py)

#### 1.1 模块职责

数据预处理是整个机器学习流水线的第一道关卡，负责将原始、嘈杂的传感器数据清洗为模型可用的高质量输入。该模块集成了四种核心处理能力：

#### 1.2 四大核心组件

| 组件 | 类名 | 算法/方法 | 核心功能 |
|------|------|-----------|----------|
| 标准化器 | `StandardScaler` / `MinMaxScaler` | Z-Score / Min-Max归一化 | 将预紧力值缩放到统一数值区间，避免尺度差异影响模型训练 |
| 异常检测器 | `IsolationForest` | 孤立森林集成学习 | 识别偏离正常分布的异常数据点（如传感器跳变、采集错误） |
| 插值处理器 | `interpolate.interp1d` | 线性/三次/最近邻插值 + 时间间隔插值 | 处理缺失值和时间序列中的大间隔断点 |
| 平滑滤波器 | `KalmanFilter` | 卡尔曼状态估计 | 消除传感器噪声，还原预紧力的真实变化趋势 |

#### 1.3 关键实现细节

**卡尔曼滤波实现**:

```python
class KalmanFilter:
    def __init__(self, process_noise=0.01, measurement_noise=0.1):
        self.process_noise = process_noise      # Q: 过程噪声（模型对变化敏感度）
        self.measurement_noise = measurement_noise  # R: 测量噪声（传感器可信度）
        
    def update(self, measurement):
        # 预测步骤：假设状态不变
        prediction = self.estimate
        prediction_error = self.estimate_error + self.process_noise
        
        # 更新步骤：用测量值修正预测
        kalman_gain = prediction_error / (prediction_error + self.measurement_noise)
        self.estimate = prediction + kalman_gain * (measurement - prediction)
        self.estimate_error = (1 - kalman_gain) * prediction_error
        return self.estimate
```

**设计亮点**: 
- 双参数配置：`process_noise` 控制模型对状态变化的敏感度，`measurement_noise` 控制对传感器测量值的信任度
- 实时增量处理：无需缓存历史数据，适合流式计算场景

**完整处理流程** (参考 `DataPreprocessor.process` [L388-L445](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/services/preprocessing.py#L388-L445)):

```
原始数据 → 异常检测(移除异常) → 缺失值插值 → 卡尔曼平滑 → 标准化 → 模型可用数据
                          ↓
                   异常数据归档存储
```

### 3. 螺栓LSTM状态预测模型

**文件位置**: [bolt_lstm.py](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/bolt_lstm.py)

#### 3.1 模块职责

单个螺栓的时序分类模型，基于最近100条预紧力数据，判断螺栓当前处于5级状态中的哪一级，并输出置信度和推荐措施。

#### 3.2 模型架构详解

参考 `LSTMNetwork` [L55-L144](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/bolt_lstm.py#L55-L144)：

```
输入层 (batch_size, 100, 2)
    │
    ▼
LSTM层1 ── 128个隐藏单元，单向，batch_first=True
    │  输出: (batch_size, 100, 128)
    ▼
Dropout层1 ── 丢弃率20%，防止过拟合
    │
    ▼
LSTM层2 ── 64个隐藏单元，单向
    │  输出: (batch_size, 100, 64)
    ▼
Dropout层2 ── 丢弃率20%
    │
    ▼
取最后时间步 [:, -1, :] ── (batch_size, 64)
    │  思路：LSTM逐步编码序列信息，最后一步包含完整上下文
    ▼
全连接层 ── 64 → 32，ReLU激活
    │
    ▼
输出层 ── 32 → 5，Softmax输出5类概率
```

**架构设计要点**:
- **双层LSTM**：第一层捕捉短期模式，第二层捕捉长期依赖
- **递减单元数**：128→64，遵循信息瓶颈原则，逐步压缩特征
- **取最后时间步**而非平均池化：对于状态分类任务，最新状态最重要

#### 3.3 5级状态定义

| 类别ID | 状态名称 | 判定标准 | 推荐措施 |
|--------|---------|---------|---------|
| 0 | 正常 | 预紧力稳定在[400,800]区间内 | 继续正常监测 |
| 1 | 关注级预警 | 轻微波动或趋势偏移，仍在安全范围内 | 加强监测频率，记录异常特征 |
| 2 | 检查级预警 | 异常趋势持续3天以上 | 组织现场检查，制定维护方案 |
| 3 | 紧急级预警 | 松动/过载/断裂前兆 | 立即处理，评估是否停机 |
| 4 | 故障 | 预紧力骤降为0或严重超标 | 紧急停机，更换损坏部件 |

#### 3.4 训练策略详解

参考 `BoltLSTMModel.train` [L239-L377](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/bolt_lstm.py#L239-L377)：

| 超参数 | 默认值 | 说明 |
|--------|--------|------|
| 损失函数 | 加权交叉熵 | 根据类别样本数自动计算权重，解决类别不平衡 |
| 优化器 | Adam | 自适应学习率，适合非平稳目标 |
| 学习率 | 0.001 | 经典默认值 |
| Batch Size | 32 | 平衡梯度稳定性与显存占用 |
| 最大Epoch | 100 | 上限足够大，由早停机制终止 |
| 验证集划分 | 20% | 时序数据中取最后20%（非随机划分） |
| 早停耐心值 | 10 | 连续10个Epoch验证损失不下降则停止 |
| 模型保存 | 最佳验证损失 | 回滚到最优权重，避免过拟合 |

### 4. 法兰面注意力关联模型

**文件位置**: [flange_attention.py](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/flange_attention.py)

#### 4.1 模块职责

同一法兰面下的螺栓并非独立工作，它们共同承受载荷，存在联动效应（一个螺栓松动会导致邻近螺栓负载增加）。本模块通过注意力机制建模多螺栓间的关联关系，输出法兰面整体的状态评估。

#### 4.2 三大核心神经网络组件

##### 组件一：螺栓特征提取器 `BoltFeatureExtractor` [L116-L170](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/flange_attention.py#L116-L170)

为每个螺栓独立编码：

```
单螺栓序列 (seq_len, 2)
    │
    ▼
双向LSTM(64隐藏单元)
    │  输出: h_n = [正向最后状态, 反向最后状态] = (batch, 128)
    ▼
全连接层: 128 → 32，ReLU激活
    │
    ▼
螺栓特征向量 (32维)
```

##### 组件二：多头自注意力 `MultiHeadSelfAttention` [L38-L113](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/flange_attention.py#L38-L113)

建模螺栓间的关联依赖：

```
输入: (batch, num_bolts, feature_dim=32)
    │
    ├──→ Q线性变换 → (batch, heads, num_bolts, head_dim)
    ├──→ K线性变换 → (batch, heads, num_bolts, head_dim)
    └──→ V线性变换 → (batch, heads, num_bolts, head_dim)
    │
    ▼
Attention(Q,K,V) = softmax(QK^T/√d) · V
    │  注意力权重矩阵揭示: 螺栓i的状态受螺栓j的影响程度
    ▼
多头拼接 + 输出投影
    │
    ▼
输出: (batch, num_bolts, 32) + 注意力权重 (可解释性分析用)
```

**8头注意力设计意图**:
- 头1-2：捕捉邻近螺栓的空间相关性
- 头3-4：识别对称位置的载荷平衡
- 头5-6：检测异常传播模式
- 头7-8：学习全局负载分配

##### 组件三：跨螺栓LSTM + 分类器

```
注意力输出 (batch, num_bolts, 32)
    │  将"螺栓"维度视为新的"时间"维度，建模结构关联性
    ▼
双向LSTM(64单元)
    │
    ▼
拼接双向末状态 → (batch, 128)
    │
    ▼
FC: 128 → 64 → ReLU → Dropout(0.2)
    │
    ▼
FC: 64 → 5 → Softmax
    │
    ▼
法兰面5类状态概率分布
```

#### 4.3 螺栓相关性分析

参考 `analyze_bolt_correlations` [L328-L361](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/flange_attention.py#L328-L361)：

计算皮尔逊相关系数矩阵：

```
corr_matrix[i][j] = pearsonr(bolt_i_series, bolt_j_series)

值范围: [-1, 1]
  +1.0: 完全正相关（一个升另一个也升，可能是共同温度效应）
   0.0: 无相关（独立工作）
  -1.0: 完全负相关（载荷转移，一个松另一个紧）
```

该矩阵可用于：
- 辅助注意力权重的可解释性验证
- 异常传播路径追溯
- 法兰面结构健康度评估

### 5. 贝叶斯风险评估模型

**文件位置**: [risk_model.py](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/risk_model.py)

#### 5.1 模块职责

区别于LSTM的"状态分类"，本模块提供更细粒度的 **风险量化** 能力。输出1-10分的风险评分（分数越高越安全），并识别具体风险因素，生成可操作的改善建议。

#### 5.2 五维风险评分体系

参考 `assess_risk` [L266-L332](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/risk_model.py#L266-L332)，综合5个维度进行加权融合：

| 评分维度 | 权重 | 评估指标 | 得分0分情形 | 得分1分情形 |
|---------|------|---------|------------|------------|
| 均值偏离度 | 25% | 预紧力均值距离正常区间中心的距离 | 完全偏离正常范围 | 恰好位于中心 |
| 波动性 | 20% | 变异系数 CV = std/mean | 剧烈波动(CV>0.2) | 极其稳定(CV<0.05) |
| 变化趋势 | 20% | 线性拟合斜率绝对值 | 快速升降趋势 | 完全平坦 |
| 极值风险 | 15% | 是否存在极端值/骤降 | 存在骤降或严重超标 | 所有值安全 |
| LSTM预测 | 20% | LSTM输出的5类概率加权 | 预测为故障 | 预测为正常 |

**加权融合公式**:

```
加权分数 = 0.25 × S_deviation 
          + 0.20 × S_volatility 
          + 0.20 × S_trend 
          + 0.15 × S_extreme 
          + 0.20 × S_lstm

最终评分(1-10) = round(加权分数 × 9 + 1, 1)
```

#### 5.3 风险等级映射

| 评分区间 | 风险等级 | 行动指南 |
|---------|---------|---------|
| 8.0 ~ 10.0 | 低风险（绿区） | 正常监测，按计划维护 |
| 4.0 ~ 7.9 | 中风险（黄区） | 提高监测频率，制定预防性维护计划 |
| 1.0 ~ 3.9 | 高风险（红区） | 立即安排现场检查，必要时停机 |

#### 5.4 诊断与建议生成机制

**风险因素识别**参考 `_identify_risk_factors` [L334-L365](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/risk_model.py#L334-L365)：

自动识别触发风险的根本原因，例如：
- `"预紧力均值偏离正常范围"` → 需要检查是否发生整体松动
- `"预紧力波动过大"` → 可能存在交变载荷或传感器干扰
- `"存在极端预紧力值"` → 可能发生瞬间过载或断裂前兆
- `"LSTM模型预测存在紧急风险"` → 深度学习模型检测到危险模式

**分层推荐措施**参考 `_generate_recommendations` [L389-L428](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/risk_model.py#L389-L428)：

根据风险等级输出基础建议，再叠加具体因素的专项建议：

```
高风险基础建议:
  1. 立即安排现场检查
  2. 评估是否需要临时停机
  3. 准备备用螺栓和工具
```

### 6. Prophet月度趋势预测模型

#### 6.1 模块职责

LSTM模型解决的是"**当前状态如何**"，而Prophet模型解决的是"**未来30天会发生什么**"。基于Facebook开源的Prophet时间序列库，进行长期趋势外推和异常时段预测。

#### 6.2 模型配置

参考 `_create_model` [L87-L105](file:///Users/zhangchengcheng/work/ai-project/Trea/solo0605/label-00205/backend/app/models/prophet_forecaster.py#L87-L105)：

#### 6.3 备选降级方案

当Prophet库不可用时，自动降级为 **线性趋势外推 + 残差标准差置信区间**：

```python
# 线性拟合历史数据
slope, intercept = np.polyfit(x, y, 1)

# 预测未来值
predicted = slope * future_x + intercept

# 基于历史残差估计不确定性
std_residual = np.std(y - (slope * x + intercept))
lower_bound = predicted - 2 * std_residual  # 95%置信下界
upper_bound = predicted + 2 * std_residual  # 95%置信上界
```

## 第三部分：项目潜在风险分析与防范建议

### 风险一：数据质量与标签质量风险

#### 风险描述

| 风险点 | 严重程度 | 影响范围 | 详细说明 |
|--------|---------|---------|---------|
| **弱监督标签不准确** | 🔴 高 | 所有LSTM模型 | 训练标签由固定阈值规则生成（参考 `_generate_labels`），非人工标注。若阈值设置不当或存在复杂异常模式，模型学习到的是"规则"而非"真实故障模式" |
| **原始训练数据不足** | 🟠 中高 | 新部署螺栓 | 训练数据不足100条时直接跳过训练，使用简单规则判断替代深度学习预测 |
| **孤立森林参数固定** | 🟡 中 | 预处理阶段 | contamination固定为10%，若真实异常率远高于/低于此值，会造成大量误删或漏删 |
| **数据分布漂移** | 🔴 高 | 长期运行后 | 设备老化/季节变换导致预紧力统计特性逐渐变化，旧模型不再适用 |
| **数据泄露风险** | 🟡 中 | 模型评估 | 训练/验证集仅做简单尾切20%，未使用时间序列交叉验证，可能泄露未来信息 |

#### 防范建议

1. **引入人工标注闭环**：建立人工审核平台，对高置信度预测结果进行抽样标注，逐步用真实标签替换规则标签
2. **动态阈值学习**：基于历史数据的分布统计（如3σ原则）自动计算每个螺栓的个性化正常区间，而非全局400-800kN
3. **定期数据审计**：调度器增加数据分布监控任务，每月计算KS检验检测分布漂移，触发自动重训
4. **时序交叉验证**：使用TimeSeriesSplit替换简单尾切，多折验证取平均性能
5. **异常率自适应**：预处理模块增加异常率监控，偏离预设范围时告警并动态调整孤立森林参数

### 风险二：模型架构与泛化能力风险

#### 风险描述

| 风险点 | 严重程度 | 影响范围 | 详细说明 |
|--------|---------|---------|---------|
| **模型数量爆炸** | 🟠 中高 | 大规模部署 | 每个螺栓/法兰面一个模型（参考需求第8点），若有10,000个螺栓则需10,000个.pt文件，存储/加载/维护成本高 |
| **LSTM序列长度硬编码** | 🟡 中 | 螺栓预测 | 固定100步输入，不同采样频率（秒级vs分钟级）的时间覆盖范围差异巨大 |
| **法兰面螺栓数量限制** | 🟡 中 | 法兰面预测 | max_bolts硬编码为20，大型法兰面可能超过此限制，多余螺栓被截断忽略 |
| **贝叶斯权重固定** | 🟡 中 | 风险评分 | prior_weights为硬编码常量（偏离25%、波动20%...），非真正贝叶斯推理，无法根据数据更新 |
| **Prophet不可用降级** | 🟡 中 | 月度预测 | Prophet库缺失时降级为线性预测，无法捕捉季节性和突变点，精度显著下降 |
| **缺少不确定性估计** | 🟠 中高 | LSTM推理 | 仅输出Softmax置信度，未考虑模型本身的认知不确定性（Epistemic Uncertainty） |

#### 防范建议

1. **分层模型策略**：引入"全局基础模型 + 螺栓个性化微调适配器（LoRA）"架构，共享大部分参数，仅保存小部分适配器权重
2. **动态序列长度**：根据采样频率自动计算所需时间窗口（如覆盖24小时所需的步数），而非固定100步
3. **螺栓数量动态扩展**：使用分组注意力或图神经网络（GNN）处理任意数量螺栓
4. **贝叶斯优化权重**：运行一段时间后，基于"评分→实际结果"的历史数据，使用贝叶斯优化更新prior_weights
5. **Prophet依赖强约束**：Docker镜像和requirements.txt中锁定Prophet版本和兼容的pystan版本，确保可用性
6. **集成Monte Carlo Dropout**：推理时启用Dropout多次前向传播，通过方差估计模型不确定性

### 风险三：预警策略与决策风险

#### 风险描述

| 风险点 | 严重程度 | 影响范围 | 详细说明 |
|--------|---------|---------|---------|
| **策略二的安全隐患** | 🔴 高 | 高风险场景 | 策略二在置信度<95%时强制输出"正常"（参考 `_apply_warning_strategy` L510-L516）。若模型对真实故障恰好只有90%置信度，将被静默为正常，可能导致严重事故 |
| **阈值缺乏数据支撑** | 🟠 中高 | 预警策略 | 70%和95%的置信度阈值、5%误报率和10%漏报率目标均为经验值，无真实数据集的ROC/PR曲线验证 |
| **缺少人机协同** | 🟡 中 | 决策流程 | 全自动报警，无人工确认环节。策略一的误报和策略二的漏报均无中间缓冲 |
| **推荐措施固定化** | 🟡 中 | 运维指导 | 推荐措施基于模板生成（如`get_recommendation`），未结合历史维修记录和故障知识库 |

#### 防范建议

1. **策略二安全护栏**：无论置信度多低，规则层面的硬阈值（如预紧力<200）必须直接报警，不允许策略二屏蔽
2. **ROC分析与阈值校准**：在验证集上绘制ROC/PR曲线，根据业务方提供的"错报成本/漏报成本"计算最优阈值
3. **三级确认机制**：
   - L1：低风险自动放行
   - L2：中风险推送人工复核
   - L3：高风险自动报警+人工确认
4. **LLM增强推荐**：结合大模型接口（参考配置中的llm.provider），传入螺栓历史、工况信息、维修记录生成定制化建议

### 风险四：系统可靠性与安全风险

#### 风险描述

| 风险点 | 严重程度 | 影响范围 | 详细说明 |
|--------|---------|---------|---------|
| **调度器状态丢失** | 🟠 中高 | 生产环境 | APScheduler使用MemoryJobStore，服务重启后所有已调度任务的状态和进度完全丢失 |
| **全局服务单例** | 🟡 中 | 并发预测 | prediction_service和training_service使用全局单例（`routes.py` L44-L61），在uvicorn多worker模式下会重复创建实例，模型缓存不共享 |
| **数据库安全配置** | 🔴 高 | 数据安全 | config.yaml中数据库密码为空字符串，CORS允许所有来源（`*`），认证默认禁用 |
| **批处理断点缺失** | 🟡 中 | 批量预测 | `_batch_predict_bolts`遍历过程中无进度记录，中途失败需从头开始，螺栓数量多时效率低 |
| **异常处理过度宽泛** | 🟡 中 | 所有接口 | 大量`except Exception`吞掉原始异常类型和堆栈，排查问题困难 |
| **无流量控制** | 🟠 中高 | API稳定性 | 无限流、熔断、降级机制，突发高并发请求可能导致OOM或服务崩溃 |

#### 防范建议

1. **持久化调度器**：使用SQLAlchemyJobStore替代MemoryJobStore，将任务状态存储到MySQL
2. **多worker适配**：使用Redis做模型缓存的分布式共享，或引入模型服务化（TorchServe/Triton）单独部署
3. **安全加固三要素**：
   - 使用环境变量注入数据库密码，禁止明文配置
   - CORS配置具体允许域名，禁用通配符
   - 启用API Key认证（配置中auth.enabled设为true）
4. **批处理进度持久化**：每成功处理N个螺栓写入进度表，重启后从断点继续
5. **精细化异常分类**：定义业务异常层次结构（如DataFormatError、ModelLoadError、PredictionTimeoutError），分别处理并保留堆栈
6. **限流与资源保护**：引入slowapi限流，GPU推理增加并发队列控制，防止显存溢出

### 风险五：工程实现与运维风险

#### 风险描述

| 风险点 | 严重程度 | 影响范围 | 详细说明 |
|--------|---------|---------|---------|
| **bolt_id类型转换隐患** | 🟠 中高 | 数据持久化 | `prediction_service.py` L570使用`int(bolt_id) if bolt_id.isdigit()`转换，若遇到非数字ID（如'B2907'）则存为NULL，外键关联失效 |
| **模型本地存储** | 🟡 中 | 分布式部署 | 模型文件存于本地磁盘，多副本部署时各节点模型不一致；磁盘损坏导致模型永久丢失 |
| **监控观测缺失** | 🟠 中高 | 生产运维 | 仅有日志输出，缺少Prometheus指标（预测QPS、延迟、准确率漂移）、缺少告警通知 |
| **GPU内存防护缺失** | 🟡 中 | GPU环境 | 配置中gpu_memory_fraction未实际使用，未设置torch.cuda.set_per_process_memory_fraction |
| **测试覆盖不足** | 🟠 中高 | 代码质量 | 测试目录存在但未验证覆盖率，核心模块如卡尔曼滤波、注意力机制缺少单元测试 |
| **训练数据泄漏** | 🟡 中 | 模型性能 | `prepare_data`中滑动窗口生成序列时，相邻序列大量重叠，导致验证集性能可能被高估 |

#### 防范建议

1. **bolt_id统一字符串处理**：数据库表bolt_id字段改为VARCHAR，移除int转换逻辑，统一字符串存储
2. **模型存储分布式化**：接入对象存储（MinIO/S3），模型版本管理器改为从对象存储读写
3. **可观测性三件套**：
   - Metrics：Prometheus暴露预测延迟、准确率、风险分布等指标
   - Tracing：OpenTelemetry追踪请求全链路（预处理→推理→写库）
   - Alerting：Grafana告警面板+钉钉/邮件通知
4. **GPU内存限制**：启动时根据gpu_memory_fraction设置`torch.cuda.set_per_process_memory_fraction`，防止CUDA OOM
5. **测试覆盖门槛**：设置80%覆盖率门禁，核心算法模块（卡尔曼、LSTM、注意力）单独测试
6. **时间序列数据划分**：序列生成后按时间边界切分训练/验证集，禁止同一时间点的数据出现在两个集合中
